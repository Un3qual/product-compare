# Future Backend Agent Guide

Use this document when building a new elixir backend. 
Assume the product premise is already defined. This guide defines implementation
conventions, architecture boundaries, and verification gates.

## 1) Non-Negotiable Conventions

- Build a modular monolith in one Phoenix app.
- PostgreSQL is the only durable datastore.
- Use `:utc_datetime_usec` everywhere (schemas, migrations, generators).
- Default relational tables to:
  - `bigint` primary key (`id`)
  - `entropy_id` (`uuidv7`) generated by Postgres
- Allow explicit UUID-primary-key exceptions only for tables like `users_tokens`.
- Persist token secrets as SHA3 hashes (`:sha3_256`), never SHA-2 and never raw.
- Keep GraphQL Relay-first:
  - `node(id:)`
  - global IDs
  - connections/edges/cursors
  - Relay mutation payload fields
- Public functions must have `@spec`.
- Keep Ecto schemas schema-only (no workflow/repo orchestration in schema modules).

## 3) Project Layout And Module Boundaries

Use the same shape with your product namespace:

```text
lib/
  your_app.ex                  # top-level core boundary (contexts public API)
  your_app/application.ex      # OTP Application + supervision tree
  your_app/
    accounts.ex
    content.ex
    ...
  your_app_schemas.ex          # top-level schema boundary exports
  your_app_schemas/
    schema.ex                  # shared schema macros
    accounts/*.ex              # schema-only modules
    content/*.ex
    ...
  your_app_gql/
    schema.ex
    relay.ex
    context.ex
    router.ex
    accounts/
      account_queries.ex
      account_mutations.ex
      account_types.ex
      account_resolver.ex
    ...
  your_app_web/
    endpoint.ex
    router.ex
    user_auth.ex
    channels/*.ex
    plugs/*.ex
test/
  support/
    data_case.ex
    conn_case.ex
    fixtures/*.ex
```

Boundary model:

- Top-level boundaries: `YourApp`, `YourApp.Application`, `YourAppWeb`, `YourAppGQL`, `YourAppSchemas`.
- Context boundaries depend on `Infra` + `Schemas`, not on web/GraphQL transport.
- Export only stable context entrypoints.

## 4) OTP Application Organization

Follow this startup pattern:

- Telemetry
- Ecto Repo
- DNS clustering
- PubSub
- Presence
- Runtime supervisors (for stateful live/session processes)
- Infra adapters (fake/dev adapters can be supervised GenServers)
- Endpoint

Rules:

- Durable facts in Postgres.
- Ephemeral runtime state in supervised processes.
- Keep runtime session processes restart strategy explicit (`:temporary` where appropriate).
- Rehydrate runtime process state from durable tables when needed.

## 5) Data Modeling And Postgres Rules

- Set generator defaults:
  - `config :your_app, ..., generators: [timestamp_type: :utc_datetime_usec]`
- Use shared schema macros:
  - `:relational` macro for bigint PK + foreign keys + microsecond timestamps
  - `:uuid_primary_key` macro only for explicit exception tables
- For every relational table:
  - `entropy_id uuid not null default uuidv7()`
  - unique index on `entropy_id`
- Keep `id` as FK target internally. `entropy_id` is additive unless explicitly promoted later.
- Require Postgres with native `uuidv7()` support (Postgres 18+).

## 6) Authentication And Token Handling

### Token storage contract

- Token rows contain:
  - DB-generated UUIDv7 `id` (for UUID exception tables)
  - `secret_hash` (binary)
  - context enum
  - optional `sent_to`, `authenticated_at`
- Transport token format:
  - `"<uuid>.<base64url(raw_secret_without_padding)>"`
- Build flow:
  - generate 32-byte random secret
  - hash with `:crypto.hash(:sha3_256, raw_secret)`
  - insert token row
  - serialize with persisted `id`
- Validate with `Plug.Crypto.secure_compare/2`.
- Never persist raw token secrets.

### Auth semantics

- Generate framework with `mix phx.gen.auth`
- Default token windows used in this repo:
  - access/session token validity: 14 days
  - refresh token validity: 30 days
  - email magic link validity: 15 minutes
  - email verification token validity: 7 days
- Maintain explicit error semantics for token auth:
  - `:invalid_token`
  - `:expired_token`
  - `:revoked_token`
- Use single-use refresh rotation:
  - consume old refresh token atomically
  - issue new access+refresh pair
- Treat revoke as idempotent at API boundary where useful.
- Record append-only auth audit events (`auth_events`) for significant auth lifecycle actions.

### GraphQL auth transport contract

- Inject GraphQL context via a plug.
- Authorization precedence:
  - `Authorization: Bearer ...` first
  - session fallback only when bearer header is absent
  - malformed/invalid bearer must not silently fall back to session
- Context keys should include:
  - `current_scope`
  - auth transport source
  - auth error reason (if any)

## 7) Absinthe + Relay Conventions

- `use Absinthe.Relay.Schema, flavor: :modern`.
- Root query must include `node field`.
- Implement strict node type resolution in `node interface`.
- Decode IDs through a single helper (like `YourAppGQL.Relay.decode_global_id/3`).
  - enforce expected node type
  - reject raw numeric IDs
- Use `node object(...)` for refetchable domain entities.
- All list surfaces should be Relay connections with deterministic sort order.
  - use `Absinthe.Relay.Connection.from_query/3` for DB-backed pagination
- Mutation conventions:
  - use `payload field ...`
  - explicit `input` and `output` blocks
  - return typed payload data + structured `errors`
  - do not use generic `successful: Boolean`
- Write actors must be derived from authenticated viewer scope, never client-supplied actor IDs.

## 8) Web/API/Realtime Conventions

- GraphQL routed through dedicated Plug router.
- GraphiQL disabled by default; enable only in explicitly non-prod config.
- REST limited to integration/webhook use-cases.
- Channels for realtime only (presence/live/chat/session events).
- Add lightweight rate limiting at edge for:
  - login/auth endpoints
  - GraphQL mutations
  - channel joins

## 9) Testing Strategy

Test layers to keep:

- Unit tests for pure decision logic.
- Context integration tests for DB + boundary behavior.
- GraphQL resolver/schema tests for contract shape and Relay behavior.
- Channel tests for realtime authorization and message flow.
- End-to-end integration tests for critical product flows.

Required test infrastructure:

- `DataCase` with SQL sandbox owner pattern.
- `ConnCase` with auth helpers.
- Fixture modules per domain in `test/support/fixtures`.

Required convention tests:

- ID convention tests (`id` + `entropy_id`, UUID exception tables).
- Token lifecycle tests (issue/authenticate/rotate/revoke + invalid/expired/revoked states).
- Relay node/global ID tests (raw IDs rejected).
- Request context tests (bearer/session precedence).

## 10) Types, Static Analysis, And CI Gates

- All public `def` must have `@spec` directly above.
- Ecto schema modules expose `@type t`.
- Centralize enum union types in namespace modules (for spec reuse).
- Keep `mix typecheck` alias that runs:
  - `mix check.typespecs --strict`
  - `mix dialyzer`
- Keep `mix precommit` alias as a hard gate:
  - compile with warnings as errors
  - unlock unused deps
  - format
  - tests
  - typecheck

## 11) Implementation Style Rules

- Keep contexts as boundary APIs, not transport adapters.
- Keep schema modules free of orchestration/business workflows.
- Separate pure decision logic from effectful coordination.
- Add concise comments for invariants and non-obvious edge-handling.
- Ensure queries used by Relay connections have stable deterministic ordering.
- Prefer idempotent behavior where API semantics need retries (e.g., unmute/revoke).

## 12) Copy/Paste Checklist For Future Agents

Before opening a PR, verify all are true:

- [ ] `mix compile` passes with no warnings.
- [ ] `mix test` passes.
- [ ] `mix typecheck` passes.
- [ ] `mix precommit` passes.
- [ ] New relational tables use `bigint id` + `entropy_id uuidv7`.
- [ ] UUID-primary-key exceptions are intentional and documented.
- [ ] Token secrets are SHA3-hashed only; raw secrets are not persisted.
- [ ] GraphQL IDs are Relay global IDs only.
- [ ] GraphQL lists use connections with stable ordering.
- [ ] Mutations are viewer-scoped and return typed payload + structured errors.
- [ ] Public functions touched in this change have `@spec`.
- [ ] New schema modules define `@type t`.
